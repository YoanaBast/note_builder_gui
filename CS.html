List vs Array

1. Python List (list)
Built-in Python type: list

Can hold elements of any type, even mixed types:

lst = [1, "hello", 3.14, True]
Dynamic size: You can append, remove, or insert elements at any time.
Flexible but slower for large numerical computations because each element is a full Python object.

lst = [1, 2, 3]
lst.append(4)
lst[0] = 10
Implementation: Array of pointers to Python objects.



2. Array (array module or numpy array)
array.array: Built-in module for arrays of same type (numbers/characters).

import array
arr = array.array('i', [1, 2, 3, 4]) # 'i' = signed int
numpy.array: Part of NumPy library, optimized for numerical operations, supports multidimensional arrays.


import numpy as np
arr = np.array([1, 2, 3, 4])
All elements must have the same type (homogeneous).

Faster for math/large data than Python lists.

Supports vectorized operations:

arr = np.array([1, 2, 3])
arr + 5 # array([6, 7, 8])
arr * 2 # array([2, 4, 6])

SOLID

1. Single Responsibility Principle (SRP) – One class, one job
    Class: Teacher
    Responsibility: Teach students
    What it shouldn’t do: Fix the school’s computers
    Why: Only one reason to change—teaching. If you make it fix computers too, changes in IT affect teaching.
Implementation:
    Keep teaching tasks in Teacher.
    Put IT-related work in a separate ITSupport class.
    One reason to change = one class.


2. Open/Closed Principle (OCP) – Open for extension, closed for modification
    Class: RemoteControl
    Behavior: Controls TV functions (volume, channels)
    Extending: Add voice control or streaming buttons without changing existing buttons
    Why: You can add new features without touching the original design.
Implementation:
    Use extensible design: e.g., allow adding new buttons/features without modifying existing ones.
    New functionality = new class or method, not editing old ones.


3. Liskov Substitution Principle (LSP) – Subtypes must work like their base type
    Class: Car
    Subclass: SportsCar
    Behavior: Drive, stop, turn
    Why: A SportsCar can replace a Car anywhere, behaving as expected. If it breaks normal car behavior, it violates LSP.
Implementation:
    Ensure all subclasses behave like the base class.
    Don’t override methods in a way that breaks expectations.
    Anyone using Car should seamlessly use SportsCar.


4. Interface Segregation Principle (ISP) – Many small interfaces > one big interface
    Class: Smartphone
    Interfaces: Calling, Camera, MusicPlayer
    Why: A person who just wants to make calls doesn’t need to implement camera or music features.
    Idea: Split big interfaces into small, focused ones.
Implementation:
    Split big interfaces into small, focused interfaces: Calling, Camera, Music.
    Classes only implement the interfaces they need, avoiding unnecessary methods.


5. Dependency Inversion Principle (DIP) – Depend on abstractions, not concretes
    Class: TV
    Depends on: PowerSource (abstract idea of electricity)
    Why: It doesn’t care if it’s a battery, plug, or solar—high-level module depends on abstraction, not concrete implementation.
Implementation:
    Depend on abstract concepts (like PowerSource) rather than concrete types.
    You can swap battery, plug, solar panel without changing the TV class.

Bubble Sort (sinking sort, exchange sort)

i and j stick together and move 1 index ahead, they swap if j is bigger
This runs multiple times until no more sorts are needed

×

Insertion Sort (“direct insertion”)

Index i starts at 0
j starts at 1
j compares to i and swaps if smaller, it does not scan the whole arr like in selection
i moves ahead until len - 1

×

Selection Sort (direct selection)

Index i stays at the start
Index j scans the whole array and finds the smallest
If it is smaller than i, swap
i is moved 1 ahead until it reaches len - 1

Time Complexities

Big-O Notation (O-notation)
    Read as: "Big O of f(n)"
    Meaning: Describes the worst-case complexity (upper bound).
    Example: If an algorithm is O(n²), it means in the worst case, the runtime grows no faster than n².

Omega Notation (Ω-notation)
    Read as: "Omega of f(n)"
    Meaning: Describes the best-case complexity (lower bound).
    Example: If an algorithm is Ω(n), it means it takes at least linear time in the best case.

Theta Notation (Θ-notation)
    Read as: "Theta of f(n)"
    Meaning: Describes the average or tight bound (both upper and lower).
    Example: If an algorithm is Θ(n log n), it means its runtime grows like n log n on average and tightly bounded there.

Binary Tree

A binary tree is a data structure where each node has at most two children:
Left child
Right child
The top node is called the root.
Nodes with no children are called leaves.
10 → root
5 → left child of 10
20 → right child of 10
Each node can have 0, 1, or 2 children.


>>> Binary Search Tree (BST)
A BST is a binary tree where for every node:
Left child ≤ parent
Right child ≥ parent
This property makes searching efficient.
Search: To find 15 → compare with root 10 → go right → 20 → go left → 15.
Time complexity: O(log n) in a balanced tree


>>> Min-Heap
A Min-Heap is a binary tree (usually implemented as an array) where:
A complete binary tree where:
Parent ≤ children
Minimum element at root
Used in priority queues.

• The smallest element is always at the root.
• Every parent node is less than or equal to its children.
• It's a complete binary tree (all levels filled except possibly the last, which is filled left to right).

In array form:
[1, 3, 5, 4, 8, 6]

"Min-Heap of size k" — What does that mean?
It means:
• You're using a Min-Heap that stores at most k elements at any time.
• This is common in algorithms where you want to track the smallest (or largest) k elements from a larger set.


>>> Max-Heap
A complete binary tree where:
Parent ≥ children
Maximum element at root
Used when you want largest element quickly.
>>><<<
BST → Organized phonebook (searchable by value)
Min-Heap → Queue where the smallest task is always on top
Max-Heap → Queue where the largest task is always on top

Merge Sort

class Solution(object):
def sortArray(self, nums):
"""
:type nums: List[int]
:rtype: List[int]
"""

if len(nums) == 1:
return nums

pivot = len(nums) // 2
left = self.sortArray(nums[:pivot])
right = self.sortArray(nums[pivot:])

result = []
i = j = 0
while i < len(left) and j < len(right):
if left[i] <= right[j]:
result.append(left[i])
i += 1
else:
result.append(right[j])
j += 1
result += left[i:]
result += right[j:]

return result

sol = Solution()
print(sol.sortArray([5,2,3,1]))

The recusrion breaks nums down to lists with 1 num inside and then after the recursion the code is run for each stack call to compare

sortArray([5,2,3,1])
LEFT
sortArray([5,2])
sortArray([5]) → returns [5]
sortArray([2]) → returns [2]
merge [5] and [2] → [2,5]
RIGHT
sortArray([3,1])
sortArray([3]) → returns [3]
sortArray([1]) → returns [1]
merge [3] and [1] → [1,3]

merge [2,5] and [1,3] → [1,2,3,5]
>> The final stage merge goes like this:
Step 0: Initial arrays
left = [2, 5]
right = [1, 3]
merged = []

Step 1: Compare 2 and 1 → take 1
merged = [1]
left = [2, 5]
right = [3]

Step 2: Compare 2 and 3 → take 2
merged = [1, 2]
left = [5]
right = [3]

Step 3: Compare 5 and 3 → take 3
merged = [1, 2, 3]
left = [5]
right = []

Step 4: Right empty → take remaining 5
merged = [1, 2, 3, 5]
no pivots

The array is split in half, recursively, and then merged.
We just talk about dividing and merging, not a pivot.

Quick Sort

class Solution(object):
def sortArray(self, nums):

if len(nums) <= 1:
return nums # Base case: already sorted

mid = len(nums) // 2
pivot = nums[mid] # Choose middle element as pivot
left = []
right = []

for i, num in enumerate(nums):
if i == mid:
continue # Skip the pivot itself
if num <= pivot:
left.append(num)
else:
right.append(num)

# Recursively sort left and right, then combine with pivot
return self.sortArray(left) + [pivot] + self.sortArray(right)


sol = Solution()
print(sol.sortArray([5, 2, 3, 1]))


Pivot: Middle element of the array.
Partition: All elements ≤ pivot go left, all > pivot go right.
Skip the pivot when partitioning.
Recursively sort left and right subarrays.
Combine: sorted left + pivot + sorted right.

Let’s see what happens if the middle element is the biggest in Quick Sort.

Suppose the array is:

[1, 2, 5, 3]
Middle element = 5 (pivot)
Step 1: Partition around pivot (5)
Left (≤ pivot): [1, 2, 3]
Right (> pivot): []
So after partitioning:

[1, 2, 3] | 5 | []


In Quick Sort, the pivot can be any element of the array. Pick any element at random. Works well in practice to avoid consistently bad cases.
import random
pivot_index = random.randint(0, len(arr)-1)
pivot = arr[pivot_index]

Closure

1. What is a Closure?
A closure is a function that “remembers” the variables from its enclosing scope, even after that outer function has finished executing.
In simpler terms: a function along with the environment in which it was created.
Closures allow you to retain state without using global variables or classes.

2. Components of a Closure
To have a closure in Python, you need:

An outer function that defines some local variables.
An inner function that uses those variables.
The outer function returns the inner function.

3. Example: Simple Closure

def outer(x):
def inner(y):
return x + y # 'x' comes from the outer scope
return inner

add_five = outer(5) # outer() returns inner()
print(add_five(10)) # Output: 15

Explanation:
x = 5 lives in the outer function.
inner() remembers x even after outer() has finished.
Calling add_five(10) adds 10 to the remembered 5 → 15.

add_five is a closure — it “closes over” x.

4. Why Closures Are Useful
Maintain state between function calls without globals.

Can be used to create factory functions (functions that generate other functions).
Enable decorators and functional programming patterns in Python.

5. Another Example: Multiplier Factory

def make_multiplier(factor):
def multiply(x):
return x * factor
return multiply

double = make_multiplier(2)
triple = make_multiplier(3)

print(double(5)) # 10
print(triple(5)) # 15

Each returned function remembers its own factor.
No global variables required.

In many languages (like JavaScript, Scheme, or Haskell), a closure is a function plus a persistent environment that keeps references to variables even after the outer function has returned.
