GitHub

>> Upload a folder to an empty GitHub repo:

cd path/to/your/folder
git init
git remote add origin https://github.com/your-username/your-repo-name.git
git add .
git commit -m "Initial commit"
git branch -M main
git push -u origin main


>> Clone a repo into an empty folder:

cd path/to/your/empty-folder
git clone https://github.com/your-username/your-repo.git


>> Remove folder after added by mistake (ex.: .idea):

git rm -r --cached .idea
git commit -m "Stop tracking .idea"
git push

>> Publish a page:

The main HTML file should be in root named index.html

1. Go to your repository on GitHub.
2. Click Settings → Pages (or scroll to the “GitHub Pages” section).
3. Under Source, select the branch: main (or master), and folder: /root.
4. Click Save.
5. GitHub will provide a URL, e.g., https://YOUR_USERNAME.github.io/python-cs-notes/
• Wait a few minutes for GitHub Pages to process.

SOLID

1. Single Responsibility Principle (SRP) – One class, one job
    Class: Teacher
    Responsibility: Teach students
    What it shouldn’t do: Fix the school’s computers
    Why: Only one reason to change—teaching. If you make it fix computers too, changes in IT affect teaching.
Implementation:
    Keep teaching tasks in Teacher.
    Put IT-related work in a separate ITSupport class.
    One reason to change = one class.


2. Open/Closed Principle (OCP) – Open for extension, closed for modification
    Class: RemoteControl
    Behavior: Controls TV functions (volume, channels)
    Extending: Add voice control or streaming buttons without changing existing buttons
    Why: You can add new features without touching the original design.
Implementation:
    Use extensible design: e.g., allow adding new buttons/features without modifying existing ones.
    New functionality = new class or method, not editing old ones.


3. Liskov Substitution Principle (LSP) – Subtypes must work like their base type
    Class: Car
    Subclass: SportsCar
    Behavior: Drive, stop, turn
    Why: A SportsCar can replace a Car anywhere, behaving as expected. If it breaks normal car behavior, it violates LSP.
Implementation:
    Ensure all subclasses behave like the base class.
    Don’t override methods in a way that breaks expectations.
    Anyone using Car should seamlessly use SportsCar.


4. Interface Segregation Principle (ISP) – Many small interfaces > one big interface
    Class: Smartphone
    Interfaces: Calling, Camera, MusicPlayer
    Why: A person who just wants to make calls doesn’t need to implement camera or music features.
    Idea: Split big interfaces into small, focused ones.
Implementation:
    Split big interfaces into small, focused interfaces: Calling, Camera, Music.
    Classes only implement the interfaces they need, avoiding unnecessary methods.


5. Dependency Inversion Principle (DIP) – Depend on abstractions, not concretes
    Class: TV
    Depends on: PowerSource (abstract idea of electricity)
    Why: It doesn’t care if it’s a battery, plug, or solar—high-level module depends on abstraction, not concrete implementation.
Implementation:
    Depend on abstract concepts (like PowerSource) rather than concrete types.
    You can swap battery, plug, solar panel without changing the TV class.

Bubble Sort (sinking sort, exchange sort)

i and j stick together and move 1 index ahead, they swap if j is bigger
This runs multiple times until no more sorts are needed

×

Insertion Sort (“direct insertion”)

Index i starts at 0
j starts at 1
j compares to i and swaps if smaller, it does not scan the whole arr like in selection
i moves ahead until len - 1

×

Selection Sort (direct selection)

Index i stays at the start
Index j scans the whole array and finds the smallest
If it is smaller than i, swap
i is moved 1 ahead until it reaches len - 1

Time Complexities

Big-O Notation (O-notation)
    Read as: "Big O of f(n)"
    Meaning: Describes the worst-case complexity (upper bound).
    Example: If an algorithm is O(n²), it means in the worst case, the runtime grows no faster than n².

Omega Notation (Ω-notation)
    Read as: "Omega of f(n)"
    Meaning: Describes the best-case complexity (lower bound).
    Example: If an algorithm is Ω(n), it means it takes at least linear time in the best case.

Theta Notation (Θ-notation)
    Read as: "Theta of f(n)"
    Meaning: Describes the average or tight bound (both upper and lower).
    Example: If an algorithm is Θ(n log n), it means its runtime grows like n log n on average and tightly bounded there.

Binary Tree

A binary tree is a data structure where each node has at most two children:
Left child
Right child
The top node is called the root.
Nodes with no children are called leaves.
10 → root
5 → left child of 10
20 → right child of 10
Each node can have 0, 1, or 2 children.


>>> Binary Search Tree (BST)
A BST is a binary tree where for every node:
Left child ≤ parent
Right child ≥ parent
This property makes searching efficient.
Search: To find 15 → compare with root 10 → go right → 20 → go left → 15.
Time complexity: O(log n) in a balanced tree


>>> Min-Heap
A Min-Heap is a binary tree (usually implemented as an array) where:
A complete binary tree where:
Parent ≤ children
Minimum element at root
Used in priority queues.

• The smallest element is always at the root.
• Every parent node is less than or equal to its children.
• It's a complete binary tree (all levels filled except possibly the last, which is filled left to right).

In array form:
[1, 3, 5, 4, 8, 6]

"Min-Heap of size k" — What does that mean?
It means:
• You're using a Min-Heap that stores at most k elements at any time.
• This is common in algorithms where you want to track the smallest (or largest) k elements from a larger set.


>>> Max-Heap
A complete binary tree where:
Parent ≥ children
Maximum element at root
Used when you want largest element quickly.
