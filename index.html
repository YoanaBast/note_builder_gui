SyntaxError: f-string: expecting '}'

Use single quotes inside double-quoted f-strings:

f"Hello, {'John'}"

Or use double quotes inside single-quoted f-strings:

f'Hello, {"John"}'

Python gets confused when it sees the same quote type inside and outside — it can’t tell where the string ends.
Use single quotes inside double-quoted f-strings:

Rule applies even in {}, ex:
f"You've ordered pizza {self.name} prepared with {", ".join(f'{i}: {q}' for i, q in self.ingredients.items())} and the price will be {self.price}lv."
(this will raise the error still)

string formatting for numbers

:.2f
________________________________
    n = 35.75432345
    formatted_number = "%.2f" % n
    print(formatted_number)
    # Output: '35.75'
    # it is a str now

    price = 3.14159
    print(f"Price: {price:.2f}")
    # Output: Price: 3.14
    # all is 1 str


:02d
________________________________
    class Time:
        def __init__(self, hours):
            self.hours = hours

        def formatted_hours(self):
            return f"{self.hours:02d}"

    t = Time(5)
    print(t.formatted_hours())
    # Output: 05


:02d → the format specification:
0 → pad with zeros
2 → width = 2 characters
d → integer (decimal)

Example:
    hours = 5
    print(f"{hours:02d}")
    # Output: 05

    hours = 12
    print(f"{hours:02d}")
    # Output: 12

So :02d means:
print as a 2-digit integer, left-padded with zeros if necessary.

# the result is always a string, no matter what type self.hours is.

unpacking operator *

The * in print(*fruits, sep=', ') is the unpacking operator.

Without *:
fruits = ['apple', 'banana', 'cherry']
print(fruits)
# Output: ['apple', 'banana', 'cherry']
# It prints the list as a list, with brackets and quotes.

With *:
print(fruits)
# Output: apple banana cherry

# The * unpacks the list, so print receives the items as if you had written:
print('apple', 'banana', 'cherry')

Sets

{} → empty dictionary, not set.
set() → empty set.
{1,} → a set with one element (the comma is optional but allowed for single-element sets).

1. Union (| or .union())
    A = {1, 2, 3}
    B = {3, 4, 5}

    A | B # {1, 2, 3, 4, 5}
    A.union(B) # {1, 2, 3, 4, 5}

2. Intersection (& or .intersection())
    A = {1, 2, 3}
    B = {3, 4, 5}

    A & B # {3}
    A.intersection(B) # {3}

3. Difference (- or .difference())
    A = {1, 2, 3}
    B = {3, 4, 5}

    A - B # {1, 2}
    B - A # {4, 5}
    A.difference(B) # {1, 2}

4. Symmetric Difference (^ or .symmetric_difference())
    A = {1, 2, 3}
    B = {3, 4, 5}

    A ^ B # {1, 2, 4, 5}
    A.symmetric_difference(B) # {1, 2, 4, 5}

5. Subset (<=, <) & Superset (>=, >)
    A = {1, 2}
    B = {1, 2, 3}

    A <= B # True (A is a subset of B)
    B >= A # True (B is a superset of A)

    print(A < B) # True
    print(B > A) # True

    A.issubset(B) # True
    B.issuperset(A) # True

>> Methods

s = {1, 2, 3}

The add() method only takes a single element at a time. If you try to pass a list or another iterable, it will treat the whole thing as one element
    s.add(4) # {1,2,3,4}
    s.add([5, 6]) # adds the *list itself* as one element
    print(s) # {1, 2, 3, 4, [5, 6]} <-- note the list inside

The update() method lets you add multiple elements to a set at once, and you can pass it any iterable — like a list, tuple, or even another set. It adds all the elements that aren’t already in the set.
    s.update([3,5]) # {1,2,3,4,5}

If you use remove(x) on a set and x is not in the set, Python will raise a KeyError.
    s.remove(2) # {1,3,4,5}

If you want to avoid an error when the element might not exist, use discard(x) instead:
    s.discard(10) # {1,3,4,5} (no error)

set.pop() does not remove the “last” element because sets in Python are unordered. It removes and returns an arbitrary element—which element you get is unpredictable.
    elem = s.pop() # removes and returns an arbitrary element

clear() removes all elements from the set, leaving it empty. It doesn’t return anything.
    s.clear() # s becomes set()

>> Reference object:

### shallow copy
    list_a = [1, 2, 3]
    list_b = list_a[:]
    list_b[0] = 5
    print(list_a) # Output: [1, 2, 3]
`list_b = list_a[:]` creates a shallow copy of `list_a`. `list_b` is now a new object, so modifying it does not affect the original `list_a`.


### reference
    list_b = list_a
    list_b[0] = 5
    print(list_a) # Output: [5, 2, 3]
list_b would be just another reference to the same list object. Modifying list_b would also change list_a


### shallow copy - nested
    list_a = [[1, 2], [3, 4]]
    list_b = list_a[:] # shallow copy
    list_b[0][0] = 9
    print(list_a) # Output: [[9, 2], [3, 4]]
Shallow copies only copy the top-level list. If the list contains nested lists or mutable objects, those inner objects are shared. So, changes to nested objects will affect the original, even with a shallow copy.

.split()

.split() is a string method.

It splits a string into a list of substrings based on a separator.

By default, the separator is any whitespace (spaces, tabs, newlines).

    text = "Hello world Python"
    words = text.split()
    print(words)
    # Output: ['Hello', 'world', 'Python']


Using a custom separator
You can pass a string as the separator:

    text = "apple,banana,cherry"
    fruits = text.split(",")
    print(fruits)
    # Output: ['apple', 'banana', 'cherry']


Limiting the number of splits
You can also limit how many splits are made using the maxsplit argument:
    text = "one two three four"
    print(text.split(maxsplit=2))
    # Output: ['one', 'two', 'three four']

.remove(), .pop(), .popitem()

>>> .pop()

Works on lists (and also dictionaries, but differently).
Removes and returns an element at a given index. If no index is given, it removes the last element at (index = -1)

lst = [10, 20, 30, 40]

# Pop last element
x = lst.pop()
print(x) # 40
print(lst) # [10, 20, 30]

# Pop element at index 1
y = lst.pop(1)
print(y) # 20
print(lst) # [10, 30]

>>> .pop() with dictionaries:

d = {'a': 1, 'b': 2}
value = d.pop('a')
print(value) # 1
print(d) # {'b': 2}
For dicts, .pop(key) removes the key and returns its value.


>>> .popitem()

Removes the last inserted key-value pair (or first if last=False).
Designed specifically for “pop the last entry” without needing the key:
d = {'a': 1, 'b': 2, 'c': 3}

# Remove the first inserted item
first_item = d.popitem(last=False)
print(first_item) # ('a', 1)
print(d) # {'b': 2, 'c': 3}

# Remove the last inserted item
last_item = d.popitem() # last=True by default
print(last_item) # ('c', 3)
print(d) # {'b': 2}

In Python 3.7+, dictionaries preserve insertion order, so you can pop the “last key” using .popitem(). The reason is that .pop() in a dict requires you to specify a key. Unlike lists, dictionaries aren’t indexed by position, they are keyed collections. You cannot just call .pop() with no arguments like you do for a list, because “the last item” isn’t inherently known to the method—it’s keyed, not indexed.


.remove()
Works only on lists.
Removes the first occurrence of a specific value.
Does not return the value.
lst = [10, 20, 30, 20]
lst.remove(20)
print(lst) # [10, 30, 20]
# Removes the first 20, not all of them.

To remove all occurrences:
    lst = [1, 2, 3, 2, 4, 2]
    while 2 in lst:
        lst.remove(2)
    print(lst) # [1, 3, 4]

    lst = [1, 2, 3, 2, 4, 2]
    lst = [x for x in lst if x != 2]
    print(lst) # [1, 3, 4]

    for books in self.rented_books.values():

The trailing comma in the assignment (...), = ... means you expect exactly one item from the dict.
(curr_position, coor) unpacks that single (key, value) pair into:

    my_dict = {'apple': 3, 'banana': 1, 'orange': 2}

SOLID

1. Single Responsibility Principle (SRP) – One class, one job
    Class: Teacher
    Responsibility: Teach students
    What it shouldn’t do: Fix the school’s computers
    Why: Only one reason to change—teaching. If you make it fix computers too, changes in IT affect teaching.
Implementation:
    Keep teaching tasks in Teacher.
    Put IT-related work in a separate ITSupport class.
    One reason to change = one class.


2. Open/Closed Principle (OCP) – Open for extension, closed for modification
    Class: RemoteControl
    Behavior: Controls TV functions (volume, channels)
    Extending: Add voice control or streaming buttons without changing existing buttons
    Why: You can add new features without touching the original design.
Implementation:
    Use extensible design: e.g., allow adding new buttons/features without modifying existing ones.
    New functionality = new class or method, not editing old ones.


3. Liskov Substitution Principle (LSP) – Subtypes must work like their base type
    Class: Car
    Subclass: SportsCar
    Behavior: Drive, stop, turn
    Why: A SportsCar can replace a Car anywhere, behaving as expected. If it breaks normal car behavior, it violates LSP.
Implementation:
    Ensure all subclasses behave like the base class.
    Don’t override methods in a way that breaks expectations.
    Anyone using Car should seamlessly use SportsCar.


4. Interface Segregation Principle (ISP) – Many small interfaces > one big interface
    Class: Smartphone
    Interfaces: Calling, Camera, MusicPlayer
    Why: A person who just wants to make calls doesn’t need to implement camera or music features.
    Idea: Split big interfaces into small, focused ones.
Implementation:
    Split big interfaces into small, focused interfaces: Calling, Camera, Music.
    Classes only implement the interfaces they need, avoiding unnecessary methods.


5. Dependency Inversion Principle (DIP) – Depend on abstractions, not concretes
    Class: TV
    Depends on: PowerSource (abstract idea of electricity)
    Why: It doesn’t care if it’s a battery, plug, or solar—high-level module depends on abstraction, not concrete implementation.
Implementation:
    Depend on abstract concepts (like PowerSource) rather than concrete types.
    You can swap battery, plug, solar panel without changing the TV class.

Time Complexities

Big-O Notation (O-notation)
    Read as: "Big O of f(n)"
    Meaning: Describes the worst-case complexity (upper bound).
    Example: If an algorithm is O(n²), it means in the worst case, the runtime grows no faster than n².

Omega Notation (Ω-notation)
    Read as: "Omega of f(n)"
    Meaning: Describes the best-case complexity (lower bound).
    Example: If an algorithm is Ω(n), it means it takes at least linear time in the best case.

Theta Notation (Θ-notation)
    Read as: "Theta of f(n)"
    Meaning: Describes the average or tight bound (both upper and lower).
    Example: If an algorithm is Θ(n log n), it means its runtime grows like n log n on average and tightly bounded there.

setattr()

>>> What does setattr do?

Python has built-in functions like setattr to dynamically assign attributes to objects.

    setattr(self, 'red_bull_team', RedBullTeam(budget))
    is exactly the same as writing:
    self.red_bull_team = RedBullTeam(budget)

But setattr lets you do it dynamically, meaning you don’t need to hard-code "red_bull_team" — you can get that name from a variable.

    team_class = RedBullTeam
     attr_name = 'red_bull_team'
     budget = 10_000_000

    setattr(self, attr_name, team_class(budget))
It’s doing:
    self.red_bull_team = RedBullTeam(10_000_000)
    But dynamically, based on:
    • Which team you’re registering (Red Bull, Mercedes, etc.)
    • Which class to create (from a dictionary)
    • Which attribute to store it in (also from the dictionary)

Why not just use self.red_bull_team = ...?
You could — but then you'd need an if:

    if team_name == "Red Bull":
        self.red_bull_team = RedBullTeam(budget)
     elif team_name == "Mercedes":
        self.mercedes_team = MercedesTeam(budget)

With setattr, you generalize that logic — you no longer need the if.

NamedTuple (typing)

from typing import NamedTuple, Any


class Pair(NamedTuple):
key: Any
value: Any

>>What is this syntax?
    • Pair is a class that inherits from NamedTuple.
    • NamedTuple is a special kind of tuple introduced in Python's typing module (or collections module in earlier versions).
    • It allows you to create tuple-like objects with named fields, so you can access elements by name instead of position.
    • The syntax key: Any and value: Any are type annotations indicating that the key and value fields can hold any type of data (Any means any type).

What does this do practically?
You can create immutable pairs with named fields:

    p = Pair(key="name", value="Alice")
     print(p.key)
    # Output: name

     print(p.value)
    # Output: Alice

Why use NamedTuple?
    • It creates a lightweight, immutable class.
    • Gives you readable code because you access fields by name instead of by index.
    • It supports type hints for better static analysis and IDE support.


When you use NamedTuple, you don’t need to write your own __init__ method because it’s automatically generated for you by the NamedTuple base class.
    • NamedTuple creates an immutable data structure with fields you define.
    • Behind the scenes, Python generates the __init__ method that takes parameters matching the field names (key and value in your example) and assigns them.
    • So you get:

    p = Pair(key="name", value="Alice")
    # This works without you writing __init__

If you had to write it manually (for a normal class), it would look like:
    class Pair:
        def __init__(self, key, value):
         self.key = key
         self.value = value

But NamedTuple saves you this boilerplate, plus adds immutability and other tuple-like behaviors.

Binary Tree

A binary tree is a data structure where each node has at most two children:
Left child
Right child
The top node is called the root.
Nodes with no children are called leaves.
10 → root
5 → left child of 10
20 → right child of 10
Each node can have 0, 1, or 2 children.


>>> Binary Search Tree (BST)
A BST is a binary tree where for every node:
Left child ≤ parent
Right child ≥ parent
This property makes searching efficient.
Search: To find 15 → compare with root 10 → go right → 20 → go left → 15.
Time complexity: O(log n) in a balanced tree


>>> Min-Heap
A Min-Heap is a binary tree (usually implemented as an array) where:
A complete binary tree where:
Parent ≤ children
Minimum element at root
Used in priority queues.

• The smallest element is always at the root.
• Every parent node is less than or equal to its children.
• It's a complete binary tree (all levels filled except possibly the last, which is filled left to right).

In array form:
[1, 3, 5, 4, 8, 6]

"Min-Heap of size k" — What does that mean?
It means:
• You're using a Min-Heap that stores at most k elements at any time.
• This is common in algorithms where you want to track the smallest (or largest) k elements from a larger set.


>>> Max-Heap
A complete binary tree where:
Parent ≥ children
Maximum element at root
Used when you want largest element quickly.

SQL Numeric Data Types

1. INTEGER (INT)
Stores whole numbers (no decimals).
Typical range: -2,147,483,648 to 2,147,483,647 (4 bytes).
Good for things like IDs, counts, ages.
Variants: SMALLINT (2 bytes), BIGINT (8 bytes).

2. SMALLINT
Smaller range than INT.
Typical range: -32,768 to 32,767 (2 bytes).
Saves storage if you know values are small (e.g., number of items in stock, months of the year).

3. BIGINT
Larger range than INT.
Typical range: -9,223,372,036,854,775,808 to 9,223,372,036,854,775,807 (8 bytes).
Use for very large counts (e.g., social media user IDs, transaction logs).

4. DECIMAL(p, s) / NUMERIC(p, s)
Stores exact fixed-point numbers.
p = precision (total number of digits).
s = scale (digits after the decimal point).
Example: DECIMAL(10,2) can store up to 99999999.99.
Great for money, prices, financial data (no rounding errors).

5. FLOAT / REAL / DOUBLE PRECISION
Stores approximate values with floating-point representation.
FLOAT → database decides precision (often 4 bytes, ~7 digits).
DOUBLE PRECISION → higher precision (8 bytes, ~15 digits).
Faster than DECIMAL, but not always exact (can have rounding errors).
Use for scientific data, measurements, percentages, but avoid for money.

These are all floating-point numeric types, meaning they store approximate numbers (not exact). The main difference is in precision (how many digits they can represent reliably) and storage size.
REAL = small, fast, less precise.
DOUBLE PRECISION = bigger, slower, much more precise.
FLOAT(p) = SQL standard that maps to REAL or DOUBLE depending on p.

6. SERIAL / BIGSERIAL (PostgreSQL only)
Auto-incrementing integer.
SERIAL = INTEGER + auto-increment.
BIGSERIAL = BIGINT + auto-increment.
Perfect for primary keys.

Always stores exactly n characters → pads with spaces if shorter.
Example: CHAR(5)
'hi' is stored as 'hi ' (3 spaces added).

Variable-length string.

Example: VARCHAR(50) → can store between 0 and 50 characters.
More flexible than CHAR.
VARCHAR (without n) → unlimited length (same as TEXT).

Stores unlimited-length text (well, very large).

No need to define a max length.
Good for long text (comments, articles, logs).

4. NVARCHAR / NCHAR (Unicode types, mainly SQL Server / Oracle)
Like VARCHAR/CHAR, but stores Unicode characters.
Needed when working with multilingual data (Chinese, Arabic, emojis, etc.).
Each character can take more bytes (UTF-16 encoding).

5. Other Useful String Types (DB-specific)
UUID (PostgreSQL, MySQL 8.0) → special string for universally unique identifiers.
JSON / JSONB (PostgreSQL, MySQL, SQL Server) → structured text data.
ENUM (MySQL, PostgreSQL) → predefined list of string values (like "pending", "approved", "rejected").

>>> General SQL Rules for Dates & Intervals

Type awareness

You can only add/subtract intervals to/from DATE or TIMESTAMP types.
Example: DATE '2025-01-01' + INTERVAL '10 days' → 2025-01-11.

Valid units for INTERVAL
Standard SQL supports: YEAR, MONTH, DAY, HOUR, MINUTE, SECOND.

Some databases add extras like WEEK, MICROSECOND, etc.

Addition vs. subtraction

date + interval → moves forward.
date - interval → moves backward.
date1 - date2 → returns an interval (the difference).

Overflow handling (months/years)

Adding months is tricky because months have different lengths.
Example in PostgreSQL:
DATE '2025-01-31' + INTERVAL '1 month' → 2025-02-28 (it clamps to the last valid day).

MySQL might behave differently.
